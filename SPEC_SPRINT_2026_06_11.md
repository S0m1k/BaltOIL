# Спринт «Правки заказчика 2026-06-11»

Ветка: `sprint-2026-06-11-pravki` (от `mobile-sprint-2026-06`).
Источник: архив с картинками от заказчика (11 шт.) + письмо про email в чате.

## Что сделано

### Заявки (order_service + frontend)
1. **Контактное лицо для приёмки** — поля «имя» и «телефон» в форме заявки,
   карточке заявки и API (`contact_person_name/phone`).
2. **Согласование крупных заявок** — заявка клиента объёмом ≥ 3000 л создаётся
   в статусе `awaiting_manager`: водители её не видят и не могут взять;
   у менеджера кнопка «Согласовать» (→ NEW) и «Отменить»; клиент видит статус
   «На согласовании» и подсказку. Заявки, созданные менеджером, согласования
   не требуют. Вкладка «На согласовании» скрыта у водителей.
3. **Список заявок** — показывается запланированная дата отгрузки
   (`desired_date`), а не дата создания (fallback на created_at).
4. **Индикация изменений** — красный «!» в списке у водителя + в баннере
   перечислено, что именно изменилось (`pending_changed_fields`: дата, объём,
   топливо, адрес, комментарий, водитель, сумма). Каждое изменение пишется
   в историю статусов.
5. **Перенос даты водителем** — больше НЕ требует подтверждения; изменения
   клиента/менеджера подтверждает только водитель (ack очищает список полей).
6. **Карандашики** — клиент (свои заявки) и водитель (назначенные) могут
   менять топливо/объём/адрес/дату через PATCH /orders с серверной матрицей
   прав; суммы пересчитываются (fail-open при недоступности сервисов).
7. **«Отметить доставку» одним окном** — фактические литры (обязательно)
   + для физлиц отметка «оплата получена» с суммой и методом;
   `final_amount` = топливо по факту + стоимость доставки.

### Зоны доставки (delivery_service + order_service)
8. **Стоимость в рублях** — у зоны поле `delivery_price` (фикс ₽ за доставку);
   админ вводит цену вместо коэффициента, она показывается клиенту в превью
   («Доставка: N ₽») и попадает в сумму заявки. Для старых зон без цены
   действует прежняя формула (коэффициент × ставка тарифа за литр).

### Регистрация и профиль юрлица (auth_service + frontend)
9. Email при регистрации юрлица необязателен; «Контактное лицо (ФИО)» → «ФИО»;
   блоки «Банковские реквизиты» и «Дополнительно» убраны из регистрации
   (поля API оставлены для совместимости).
10. Профиль юрлица: все реквизиты (ИНН/КПП/ОГРН/юр.адрес/банк/billing email/
    адрес доставки) видны и редактируются клиентом («Редактировать реквизиты»);
    кнопка добавить/изменить email — для всех клиентов.

### Чаты (chat_service + auth_service + frontend)
11. **Фото/видео** — POST/GET `/conversations/{id}/attachments`
    (25 МБ, jpg/png/webp/gif/mp4/mov/webm, имена uuid.ext), сообщения
    `msg_type=photo|video`, рендер с авторизованной загрузкой;
    REST-сообщения публикуются в Redis-канал чата (realtime у собеседников).
12. **Папки и закрепление** — чипы Все/Рабочие/Клиенты/Личные;
    пин (`POST /conversations/{id}/pin`), закреплённые первыми.
13. **Email при первом сообщении клиента** — модал «укажите почту для
    уведомлений об ответе», сохраняется в профиль (PATCH /users/{id});
    письма об ответах уже шлёт notification_service оффлайн-пользователям.
14. **Блокировка мессенджера** (только admin) — чекбокс «ВЫКЛ мессенджер»
    в «Настройках клиента» (PATCH /users/{id}/tariff, admin-only).
    Заблокированный клиент: не пишет, не открывает переписку
    («Доступ ограничен»), не находится по номеру (кроме сотрудников).
15. **Чат с бухгалтерией для юрлиц** — `kind=client_accountant`, авто-создание
    при открытии чатов клиентом-юрлицом; у клиента кнопки «Запросить акт
    сверки» (период обязателен) и «Запросить закрывающие документы»
    (крайняя отгрузка / конкретный документ); у сотрудников быстрый ответ
    «Направили по ЭДО».

## Деплой

Миграции (идемпотентные):
- order_service: **0016** (contact_person_*, pending_changed_fields,
  enum-значение `awaiting_manager`)
- delivery_service: **004** (delivery_zones.delivery_price)
- auth_service: **0009** (client_profiles.messenger_blocked)
- chat_service: **0004** (conversation_participants.is_pinned)

Шаги:
1. `git checkout sprint-2026-06-11-pravki && git pull`
2. Пересобрать: auth, order, delivery, chat (новая зависимость
   `python-multipart` в chat_service → обязательно rebuild образа),
   force-recreate frontend.
3. Миграции прогоняются entrypoint'ами при старте (как обычно).
4. Вложения чата пишутся в `chat_service/media/chat/{conv_id}/`
   (volume `./chat_service:/app` уже смонтирован; каталог в .gitignore).

## Открытые вопросы заказчику
- Флоу согласования > 3000 л: сделано «менеджер жмёт Согласовать → заявка
  уходит водителям». Если нужен другой флоу — обсудить (заказчик предлагал
  созвон).
- При недогрузе сумма пересчитывается по фактическим литрам автоматически —
  подтвердить, что так и нужно.
- «Зоны показа» на карте — что имелось в виду? Сейчас зоны = стоимость
  доставки; зон видимости сервиса нет.
- Мобильное приложение (Flutter) пока не обновлялось под новые статусы/поля —
  отдельной задачей.
