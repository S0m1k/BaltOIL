## 1. Цель Etap 3

Нативные мобильные приложения для всех трёх ролей BaltOIL — чтобы уведомления о звонках, сообщениях и событиях заказа приходили нативно (не теряясь в email), а звонки работали как обычный телефон (полноэкранный входящий, через bluetooth, при заблокированном экране).

**Триггер для старта:** веб-версия стабилизирована (нет регулярных hotfix'ов, payment+document пайплайны в проде работают, claim-rate низкий).

## 2. Согласованные решения (lock-in 2026-05-25)

### Платформы и порядок
- **Android first**, после релиза Android — iOS.
- Минимальные версии — определит architect (предварительно: Android 8.0+ / iOS 14+).

### Архитектура
- **Тонкий клиент.** Вся бизнес-логика остаётся на бэке. Мобилка = клиент к существующим REST/WebSocket API + нативная прослойка для push/звонков/CallKit.
- **Одно приложение на все 3 роли** (клиент / менеджер / водитель). UI разворачивается по роли после логина. Не три отдельных приложения.
- Authorization — единая, существующий `auth_service`, токены долгоживущие с refresh.

### Стек (рекомендация, финал — на этапе architect)
- **React Native** — единая кодовая база Android+iOS, готовые SDK для LiveKit, FCM/APNs, CallKit/ConnectionService.
- Альтернативы для пересмотра архитектором: Flutter, полная нативка (Kotlin+Swift).
- Отказались от: PWA + WebView (слабый UX звонков, проблемы с фоном).

### Push-уведомления
- **FCM (Android) + APNs (iOS).**
- Бэк отправляет в FCM/APNs, ОС показывает нативно, клик открывает нужный экран.
- **Email-уведомления сохраняются параллельно с push** — не у всех есть приложение, юридические документы (счета/УПД) первично идут email'ом, push-доставка не гарантирована (Doze, отключённые уведомления).
- Правило: «есть установленное приложение и юзер залогинен → push первичен, email-копия по настройке пользователя; нет приложения → email как сейчас».

### Звонки
- **LiveKit SDK** (нативные SDK для Android/iOS) — выбор LiveKit на Etap 2 был сделан именно ради совместимости с мобилкой.
- **CallKit (iOS) / ConnectionService (Android)** — для нативного UX входящего звонка: полноэкранный, при заблокированном экране, через bluetooth-гарнитуру.
- Звонки между всеми ролями — менеджер↔клиент, водитель↔клиент, внутри компании.

### Чат
- Существующая WebSocket-логика бэка, мобилка только рендерит сообщения.
- **При сокете в фоне или отвалившемся сокете** — push-уведомление о новом сообщении вместо WS-события.

### Оффлайн — ограниченный scope
Оффлайн **только** для двух пайплайнов:
1. **Статусы заявок** — водитель/менеджер меняет статус заказа без связи, синхронизация при появлении сети.
2. **Оплаты** — отметка оплаты (особенно `on_delivery`) офлайн с последующей синхронизацией.

**Всё остальное требует связи:** создание заказа, чат, документы, история, звонки. Локальная БД (SQLite/Realm) только под эти два пайплайна.

**Conflict resolution:** правила определит architect. Предварительно — last-write-wins по timestamp, с пометкой о расхождении в лог для менеджера.

## 3. Прерeq в Etap 2 (LiveKit calls) — заложить заранее

Чтобы на Etap 3 не возвращаться и не переделывать бэк, при работе над Etap 2 (звонки в чате) реализовать:

- [ ] **WebSocket reconnect** — прозрачное переподключение при разрыве. Мобильный Doze mode рвёт сокеты постоянно — клиент должен переподключаться без потери сообщений.
- [ ] **Долгоживущие токены** для мобильных клиентов — refresh tokens, ротация. Web-сессии короткие, мобильные живут недели/месяцы.
- [ ] **Авторизация WebSocket по токену** в заголовке/query, **не по cookie**. Мобильные клиенты не работают с cookie.
- [ ] **Push-fallback события** — когда WS-клиент не подключён, событие должно уходить в очередь для push-отправки. На Etap 2 — заглушка (логирование), на Etap 3 — реальная отправка через FCM/APNs.
- [ ] **API-эндпоинты идемпотентны** где возможно (создание заказа, смена статуса, отметка оплаты) — мобилка может ретраить запросы при flaky-сети.

Эти пункты — часть SPEC Etap 2, не Etap 3.

## 5. Что НЕ делать на Etap 3 (boundaries)

- **Не переписывать бэк под мобилку.** Если что-то на бэке требует переделки — это знак, что надо доделать на предыдущих этапах или вынести в отдельный мини-этап.
- **Не делать offline-режим для чата, создания заказов, документов.** Только статусы и оплаты.
- **Не делать кастомный WebRTC.** Только LiveKit SDK.
- **Не делать три отдельных приложения.** Решено: одно приложение, роли по логину.
- **Не отключать email-уведомления.** Push добавляется параллельно, email остаётся.
- **Не публиковать в PlayStore/AppStore без согласования с user** — особенно водительское приложение.

## 7. Acceptance criteria для MVP Android (предварительно)

Уточняется архитектором при старте, но базовый каркас:

- [ ] Логин по существующим credentials, токен сохраняется в Keystore.
- [ ] Роль определяется при логине, UI разворачивается соответственно.
- [ ] Push-уведомления о новом сообщении в чате — приходят, клик открывает чат заказа.
- [ ] Push-уведомления о смене статуса заказа — приходят с deep-link.
- [ ] Входящий звонок отображается через ConnectionService (полноэкранный, при заблокированном экране).
- [ ] LiveKit-звонок работает (audio минимум, video — опционально для MVP).
- [ ] Водитель видит свои заказы, может менять статус — в т.ч. оффлайн с последующей синхронизацией.
- [ ] Менеджер видит список заказов, может открыть карточку клиента, видит финансы.
- [ ] Клиент видит свои заказы и историю, может писать в чат.
- [ ] Email-уведомления продолжают приходить параллельно с push (через настройки можно отключить).

iOS MVP — те же критерии, после релиза Android.
