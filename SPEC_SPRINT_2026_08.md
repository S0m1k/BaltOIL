# SPEC — Спринт 2026-08 (правки заказчика: упрощение статусов, каталог топлива, зоны доставки, SMS-вход)

**Контекст.** Заказчик 2026-06-05 прислал большой пакет правок — часть возвращает то, что ранее упрощали/удаляли (статусы, частичная доставка), часть откатывает Sprint 2026-07 (договор/POA/УПД/ТТН из выдачи документов). Ниже — согласованный с владельцем (Opus, 2026-06-05) скоуп, разложенный на 6 независимых деплоев.

**Решения, зафиксированные при планировании (Opus, 2026-06-05).**
- **Статусы заявки → 3 рабочих + отменённые:** `Новая → Принята водителем → Доставлена` + отдельная вкладка `Отменённые`. Убираем `IN_TRANSIT`, `PARTIALLY_DELIVERED`, `CLOSED` и **всю логику частичной/мульти-рейсовой доставки** (откат `c7aaf02`).
- **Нумерация** — раздельные счётчики по виду заявки, голый префикс без года: `ф{n}` (физ), `ю{n}` (юр), `л{n}` (ТТН-Л).
- **ТТН-Л** — отдельный **вид заявки**, создаёт только менеджер на конкретного водителя; видна только этому водителю + менеджерам/админам; **списывает топливо со склада**, идёт по тем же 3 статусам; **клиентских счетов не выставляет** (внутренняя).
- **Номер ТТН** при отметке «Доставлена» — обязательное текстовое поле, фиксируем; документ ТТН **не** генерим.
- **Документы** — активны только `INVOICE_PRELIMINARY` (после создания заявки) и `INVOICE_FINAL` (после доставки), **оба видны клиенту**. Остальные типы (`UPD`, `TTN`, `CONTRACT`, `POA`, legacy `INVOICE`) остаются в БД «спящими», убираются из выдачи/генерации.
- **«Основание»** в счёте: `Договор о поставке Нефтепродуктов от «{дата}»`. Строку о доставке из позиций счёта убираем; стоимость доставки заложена в сумму заявки (расчёт по зоне).
- **Порог авто-счёта:** при `volume >= 3000` л счета **не** генерятся автоматически — вместо этого уведомление менеджеру.
- **Каталог топлива** — выносим хардкод-enum в **таблицу-каталог, источник истины = `order_service`**, `delivery_service` читает справочник по internal-API. CRUD для админа. «К5» = **переименование** существующих ДТ-Л/ДТ-З в «ДТ-Л К5» / «ДТ-З К5» (коды не меняем, меняем label). Сезонный дефолт: ноябрь–апрель → зимнее, иначе летнее. В форме заявки показываем только топливо с остатком > 0.
- **Минимальный объём** — 300 л (вместо 1000).
- **Зоны доставки** — роутер в `delivery_service` (не плодим сервис). Рендер/рисование — Leaflet + leaflet-geoman, тайлы OSM, адреса/геокодинг — **DaData** (уже интегрирована). Адрес→зона через point-in-polygon. У зоны — коэффициент стоимости доставки.
- **Тарифы** — два базовых (физ/юр) + переопределение per-client (`tariff_id` остаётся); зональный коэффициент в расчёте стоимости. Volume-tiers сохраняем как есть.
- **Оплаты:** `payment_status` ∈ `{unpaid, paid}` (`partially_paid` убираем). «Ждёт оплаты» = создана и не оплачена при требовании предоплаты. Долговые заявки создаёт **только менеджер**, клиент — не может; водитель долга не видит, ему сразу «можно доставлять». Водитель проставляет оплату **только физикам** + popup с суммой после доставки; у юриков водитель денег не видит вообще. Менеджер/админ проставляют у всех. Клиенту при неоплаченной заявке — крупное визуальное оповещение «свяжитесь с менеджером».
- **Auth/SMS:** регистрация физика — только номер/имя/пароль; email дозаполняемый в ЛК с подсказкой. Вход: номер+пароль **или** номер+**SMS-код**. Сброс/восстановление пароля по SMS-коду. Провайдер — **SMSC.ru**, адаптер в `notification_service`, код в Redis с TTL. Юрики — номер и email, всё обязательно. **Админ редактирует данные всех клиентов.**
- **Склад водителю** — read-only доступ к остаткам (без прихода).
- **Перенос заявки** — смена даты и/или водителя, доступно всем ролям; уведомляем водителя; если заявка уже «Принята водителем», правки/перенос требуют **подтверждения водителем** («увидел изменения»).
- **Мониторинг машин (как Я.Карты)** — НЕ в этом спринте, в parking lot.

**Принципы.**
- BaltOIL — master only, без feature-веток.
- 6 деплоев, каждый катится независимо. Все миграции идемпотентные, по 1 на деплой.
- `.env` на проде не трогаем (только `.env.example`). Untracked на сервере не трогаем.
- Зависимости: **Д3 (зоны)** нужен до **Д4 (документы)** и **Д5 (оплаты/тарифы)** — стоимость доставки и зон. коэф. **Д2 (каталог топлива)** желателен до **Д5** (цены по типам).

## Деплой 1 — Ядро заявок: статусы, вкладки, нумерация, перенос, ТТН-Л

Цель: упростить статусы до 3+отменённые, заменить чек-боксы вкладками, раздельная нумерация, ввод номера ТТН, полное редактирование с подтверждением водителя, перенос.

### 1.1 Статусы заявки
**Файлы:** `order_service/app/models/order.py:22` (OrderStatus), `order_service/app/core/status_machine.py`.
- Новый набор `OrderStatus`: `NEW` (Новая), `ACCEPTED` (Принята водителем), `DELIVERED` (Доставлена), `CANCELLED` (Отменена).
- Удалить `IN_TRANSIT`, `PARTIALLY_DELIVERED`, `CLOSED`. Вычистить связанную логику частичной/мульти-рейсовой доставки (кумулятивный объём из `c7aaf02`), `PARTIALLY_DELIVERED`-ветки в `status_machine`, расчётах и шаблонах.
- Машина переходов: `NEW → ACCEPTED` (водитель принимает / claim), `ACCEPTED → DELIVERED` (водитель, **требуется `ttn_number`**), `NEW|ACCEPTED → CANCELLED` (менеджер/админ; кто ещё может отменять — уточнить, по умолчанию только staff).
- Миграция данных (идемпотентная, подтверждена 2026-06-05): `IN_TRANSIT → ACCEPTED`, `PARTIALLY_DELIVERED → ACCEPTED`, `CLOSED → DELIVERED`, `REJECTED → CANCELLED`.

### 1.2 Вид заявки и нумерация
**Файлы:** `order_service/app/models/order.py`, `order_service/app/services/order_number.py:28`, `order_service/app/services/order_counter.py`.
- Поле `order_kind` на заявке: `individual` / `company` / `ttn_l`. Для обычных заявок выводится из типа клиента; `ttn_l` ставится при создании менеджером с чек-боксом «ТТН-Л».
- Нумерация: голый префикс без года — `ф{n}` / `ю{n}` / `л{n}`. Раздельные атомарные счётчики на каждый kind (расширить `order_counter` ключом по kind, upsert `ON CONFLICT`).
- **Очистка существующих заявок (решение 2026-06-05, БД до прода — данные тестовые).** Старый формат `ORD-YYYY-*` уходит полностью; перенумерации нет. Миграция-очистка удаляет:
  - все существующие заявки (`orders`) и их документы (`documents`);
  - связанные рейсы (`delivery_service.trips`) и **departure**-транзакции склада (`fuel_transactions` с `type=departure` / непустым `order_id`);
  - **приходы на склад сохраняем** (`type=arrival` не привязан к заявкам); после очистки — `POST /inventory/reconcile`, чтобы `fuel_stock` пересчитался из оставшихся транзакций.
  - Счётчики `order_counter` обнуляются — новая нумерация `ф/ю/л{n}` стартует с 1.
  - Очистка выполняется отдельным **скриптом/миграцией с явным подтверждением**, не как часть авто-`upgrade head` (защита от случайного запуска на проде с реальными данными).

### 1.3 ТТН-Л
- Создаётся только менеджером, на **конкретного водителя** (`driver_id` обязателен), `order_kind=ttn_l`. Видна только этому водителю + менеджерам/админам (фильтрация в `order_service` listing + проверка доступа в `get_order`).
- Списывает топливо со склада при доставке (та же departure-транзакция, что и у обычной заявки). Клиентских счетов не выставляет (см. Д4 — пропуск генерации для `ttn_l`).

### 1.4 Назначение водителя
- Клиент создал заявку → `driver_id = NULL`, видна **всем** водителям (пул, кто первый принял).
- Менеджер создал → выбор «всем» (`driver_id=NULL`) или «конкретному» (`driver_id` задан, видна только ему).

### 1.5 Ввод номера ТТН при доставке
- При переходе `ACCEPTED → DELIVERED` водитель вводит `ttn_number` (текст, обязательно). Хранить на заявке/рейсе, показывать в карточке staff.

### 1.6 Полное редактирование + подтверждение водителя
**Файлы:** `order_service/app/routers/orders.py`, `order_service/app/services/order_service.py`.
- Менеджер/админ могут редактировать все поля заявки в любом статусе.
- Если заявка в `ACCEPTED` и менеджер/админ её редактирует или переносит → выставить флаг `pending_driver_ack=true`. Водителю — баннер «заявка изменена, подтвердите». До подтверждения — пометка в его списке. Эндпоинт `POST /orders/{id}/ack-changes` (driver) снимает флаг.

### 1.7 Перенос заявки
- Эндпоинт `POST /orders/{id}/reschedule` — смена `delivery_date` и/или `driver_id`. Доступно всем ролям (клиент — только свою; staff — любую; водитель — назначенную ему). Уведомление водителю (через `notification_service`, как остальные order-события). Перенос принятой заявки → `pending_driver_ack=true` (см. 1.6).

### 1.8 Убрать время доставки
**Файлы:** `order_service/app/models/order.py:40` (`delivery_window`), схемы, фронт.
- Удалить `delivery_window` enum и поле, убрать из формы и карточек. Миграция дропает колонку.

### 1.9 Фронтенд — вкладки вместо чек-боксов
**Файлы:** `frontend/index.html` (`#pane-orders`, чек-боксы ~145, payment-фильтр ~164).
- Убрать чек-боксы статусов. Три вкладки + отменённые: **Новые / Ждут доставки / Доставленные / Отменённые**, маппятся на `NEW / ACCEPTED / DELIVERED / CANCELLED`.
- Чек-бокс «ТТН-Л» в форме создания заявки менеджером (при выборе конкретного водителя).
- Поле ввода номера ТТН в диалоге «Доставлена». Баннер «заявка изменена — подтвердить» у водителя.

### 1.10 Acceptance
- В БД и UI только 4 статуса; старые мигрированы. Нет ни одной ссылки на частичную доставку.
- Новые заявки нумеруются `ф/ю/л{n}` раздельно. ТТН-Л видна только адресату+staff.
- Доставка без номера ТТН невозможна. Перенос/редактирование принятой заявки требует ack водителя и шлёт ему уведомление.

## Деплой 3 — Зоны доставки и стоимость

Цель: карта с зонами (админ рисует), адрес клиента→зона, зональный коэффициент в стоимости, доставка заложена в заявку.

### 3.1 Модель зон (delivery_service)
**Файлы:** новый `delivery_service/app/models/delivery_zone.py`, роутер `delivery_service/app/routers/zones.py`.
- `delivery_zones`: `id`, `name`, `polygon` (GeoJSON/массив координат), `cost_coefficient` (или базовая стоимость доставки), `is_active`.
- CRUD — admin only. Хранение полигона как JSONB.
- `POST /zones/resolve` — по адресу/координатам вернуть зону (point-in-polygon на бэке).

### 3.2 Геокодинг адреса
- Автокомплит адреса клиента через **DaData** (suggestions/geolocate) — переиспользуем существующую интеграцию (`auth_service` lookup ИНН → вынести/повторить клиент DaData для адресов). Адрес → координаты → `resolve` зоны.

### 3.3 Фронт — карта
**Файлы:** `frontend/index.html` (admin-вкладка «Зоны» + блок в форме заявки).
- Админ: Leaflet + leaflet-geoman, тайлы OSM, рисование/редактирование полигонов, задание коэффициента, сохранение в `/zones`.
- Клиент при создании заявки: карта с зонами + поле адреса (DaData-автокомплит). Показ выбранной зоны и рассчитанной стоимости доставки.

### 3.4 Стоимость доставки в заявке
**Файлы:** `order_service/app/services/pricing_service.py:49`.
- При создании заявки: `order_service` резолвит зону (internal-вызов в `delivery_service`) → коэффициент → стоимость доставки **внутри суммы заявки** (отдельной строкой в счёте НЕ выводится, см. Д4).

### 3.5 Acceptance
- Админ рисует зоны на карте и задаёт коэффициенты. Клиент вводит адрес, видит зону и стоимость доставки, заложенную в сумму.
- **Адрес вне всех зон:** заявка не блокируется — стоимость доставки выставляет менеджер вручную (поле «стоимость доставки» в заявке доступно staff; клиенту показывается «уточняется менеджером»).

## Деплой 4 — Документы (2 счёта)

Цель: оставить только предв./финальный счёт, оба клиенту, «основание», без строки доставки, порог 3000 л.

### 4.1 Сужение типов
**Файлы:** `order_service/app/models/document.py:10`, `order_service/app/routers/documents.py`, `order_service/app/services/document_service.py`, `document_export.py`, шаблоны `order_service/app/templates/invoice.html`.
- Активны только `INVOICE_PRELIMINARY`, `INVOICE_FINAL`. `UPD/TTN/CONTRACT/POA/INVOICE(legacy)` — оставить в enum/БД, убрать из генерации, выдачи и UI (спящие).
- Оба счёта **видны клиенту** (текущая логика доступа к своим документам — расширить выдачу обоих типов клиенту).

### 4.2 Триггеры генерации
- `INVOICE_PRELIMINARY` — авто после **создания заявки**. `INVOICE_FINAL` — авто после **`DELIVERED`**.
- **Порог:** если `order.volume >= 3000` л — **не** генерим автоматически; шлём уведомление менеджеру (через `notification_service`). Менеджер может выставить вручную.
- Для `order_kind=ttn_l` счета не генерим вовсе.

### 4.3 Содержимое счёта
**Файлы:** `order_service/app/templates/invoice.html`, `document_export.py`.
- «Основание»: `Договор о поставке Нефтепродуктов от «{дата}»`. Источник даты (решение 2026-06-05): предварительный счёт → **дата создания заявки**; финальный счёт → **дата оплаты** (fallback — дата доставки, если оплата ещё не проставлена).
- **Убрать строку о доставке** из позиций; стоимость доставки уже включена в сумму заявки (Д3).

### 4.4 Acceptance
- Клиент видит оба счёта. Предв. — при создании, финальный — после доставки. Заявка ≥3000 л → счета не выставлены автоматически, менеджер получил уведомление. В счёте корректное «основание», нет отдельной строки доставки. По ТТН-Л счетов нет.

## Деплой 5 — Тарифы и оплаты

Цель: два базовых тарифа физ/юр + per-client, статусы оплаты, долговые заявки, права на проставление оплаты, оповещения.

### 5.1 Тарифы
**Файлы:** `order_service/app/models/tariff.py`, `pricing_service.py:49`, seed.
- Два дефолтных тарифа: `individual` и `company`. Клиент по типу получает соответствующий по умолчанию; `ClientProfile.tariff_id` позволяет переопределить.
- Зональный коэффициент (Д3) участвует в расчёте стоимости. Volume-tiers сохраняем.

### 5.2 Платёжные статусы
**Файлы:** `order_service/app/models/order.py:105` (`payment_status`).
- `payment_status` ∈ `{unpaid, paid}` — убрать `partially_paid` (миграция: `partially_paid → unpaid`, подтвердить).
- «Ждёт оплаты» = `unpaid` при требовании предоплаты (флаг `payment_required`/неевляется долговой).

### 5.3 Долговые заявки
- Поле `allow_delivery_unpaid` (долг). Ставит **только менеджер** при создании; клиент создать долговую заявку не может. Водитель этот флаг **не видит** — при долге ему показывается «можно доставлять» как при оплаченной.

### 5.4 Права на оплату
- Водитель проставляет оплату **только** заявкам физлиц (`order_kind=individual`); после отметки «Доставлена» физику — **popup с суммой**. У юрлиц водитель денег не видит вообще (ни суммы, ни статуса оплаты).
- Менеджер/админ проставляют оплату у всех.

### 5.5 Оповещение клиента
**Файлы:** `frontend/index.html` (клиентская карточка заявки).
- При `unpaid` + требуется оплата — крупное визуальное оповещение «Заявку необходимо оплатить. Свяжитесь с менеджером».

### 5.6 Acceptance
- Два тарифа применяются по типу клиента, переопределяемы. Водитель видит деньги только у физиков (+popup), у юриков — нет; долг ему не виден. Долговую заявку создаёт только менеджер. Клиент видит крупное оповещение о неоплате.

## Деплой 6 — Auth / SMS

Цель: упрощённая регистрация физика, вход по SMS-коду, сброс по SMS, email дозаполняемый, админ редактирует клиентов.

### 6.1 SMS-провайдер (SMSC.ru)
**Файлы:** новый адаптер `notification_service/app/services/sms_service.py`, internal-эндпоинт `POST /internal/sms/send-code`. Конфиг в `.env.example` (логин/пароль/sender SMSC).
- Отправка SMS-кода через SMSC.ru HTTP API. Код генерится и хранится в **Redis с TTL** (напр. 5 мин), ключ по телефону/цели (login/reset). Rate-limit на отправку.

### 6.2 Регистрация физика
**Файлы:** `auth_service/app/routers/auth.py:18` (`/register/individual`), `auth_service/app/models/client_profile.py:12`.
- Поля только: телефон, имя, пароль. Email **необязателен** (дозаполняется в ЛК).
- Телефон — основной логин-идентификатор для физика (уникальность).

### 6.3 Вход
- Два способа: телефон+пароль **или** телефон+SMS-код. Эндпоинты `POST /auth/login` (как есть) и `POST /auth/login/request-code` + `POST /auth/login/verify-code`.

### 6.4 Сброс/восстановление пароля
- `POST /auth/password/request-code` (по телефону) → SMS-код → `POST /auth/password/reset` (код + новый пароль).

### 6.5 Email дозаполняемый
**Файлы:** ЛК-форма профиля, `auth_service` update-эндпоинт.
- Поле email в ЛК с подсказкой «Заполните email, чтобы получать уведомления». Уведомления физику идут на email только если он задан (иначе — push/SMS-канал по мере наличия).

### 6.6 Юрики
- Регистрация/профиль юрика — телефон **и** email обязательны (остальные поля как сейчас, DaData по ИНН сохраняем).

### 6.7 Админ редактирует всех клиентов
**Файлы:** `auth_service/app/routers` (admin-эндпоинты), фронт admin-карточка клиента.
- Админ может редактировать любые данные любого клиента (физ/юр), включая профильные поля и контакты.

### 6.8 Acceptance
- Физик регистрируется по номеру/имени/паролю, входит паролем или SMS-кодом, сбрасывает пароль по SMS. Email необязателен при регистрации, дозаполняется в ЛК. Юрик — телефон+email обязательны. Админ правит данные любого клиента. SMS уходят через SMSC.ru, коды живут в Redis с TTL.
